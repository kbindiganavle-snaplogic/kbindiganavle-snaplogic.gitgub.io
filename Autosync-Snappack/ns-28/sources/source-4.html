51 /**
52  * Kerberos utilities.
53  *
54  * FIXME: for the moment this is a static class. We need to refactor this so that it's an
55  * injected interface in order to facilitate testing - we don't want tests to require a
56  * fully configured Kerberos environment.
57  *
58  * Kerberos notes
59  *
60  * Kerberos requires a standard /etc/krb5.conf file (or equivalence). If it is not possible,
61  * or the user wants to override those values for some reason, two system properties can be
62  * used in its place:
63  *
64  * <ul>
65  * <li>java.security.krb5.realm</li>
66  * <li>java.security.krb5.kdc</li>
67  * </ul>
68  *
69  * There are two useful system properties for debugging
70  *
71  * <ul>
72  * <li>sun.security.krb5.debug=true</li>
73  * <li>java.security.debug=gssloginconfig,configfile,configparser,logincontext</li>
74  * </ul>
75  *
76  * Hadoop-based Kerberos (Hive, Impala, others?) requires the location of the Hadoop
77  * configuration files. Note: we only need the configuration files, not a full Hadoop
78  * installation. The location of these files is specified via a system property:
79  *
80  * <ul>
81  * <li>hadoop.home.dir</li>
82  * </ul>
83  *
84  * For convenience if this value is not set but HADOOP_HOME is set then we'll set the property.
85  * We do not attempt to set it to the default value of /etc/hadoop.
86  *
87  * As an alternative the following system properties can be set explicitly in getJaasDataSource(),
88  * getHadoopDataSource(), and (probably) {@link PrivilegedDataSourceAction#run()}.
89  *
90  * <ul>
91  * <li>hadoop.security.authentication=kerberos</li>
92  * <li>hadoop.security.authorization=true</li>
93  * <li>hive.server.authentication=KERBEROS</li>
94  * <li>hive.server2.authentication=KERBEROS</li>
95  * </ul>
96  *
97  * Note that this will cause complications of the JVM requires a mixture of Kerberos- and
98  * non-Kerberos-authenticated connections.
99  *
100  * The location of the 'kinit' command should be specified by a system property. (It defaults
101  * to 'kinit' if unspecified but that it may not be on the classpath.
102  *
103  * <ul>
104  * <li>hadoop.kerberos.kinit.command</li>
105  * </ul>
106  *
107  * Finally with Cloudera, at least, this system property must be set.
108  *
109  * <ul>
110  * <li>javax.security.auth.useSubjectCredsOnly=false</li>
111  * </ul>
112  *
113  * (from https://community.cloudera.com/t5/Batch-SQL-Apache-Hive/
114  * Hive-JDBC-client-error-when-connecting-to-Kerberos-Cloudera/td-p/30829)
115  *
116  * Notes on locks and system properties (per SNAP-3018 review)
117  *
118  * This is a Hard Problem (TM) since we can't ensure the system properties are in a
119  * known state and we don't have any way to prevent other threads from modifying the
120  * system properties under us even if we have locks. (Ditto we can't be sure that
121  * we don't change the system property under a different thread when we reset it to
122  * the original value.) All we can do is ensure that our code is single-threaded
123  * through the sensitive block of code.
124  *
125  * It's not an ideal solution but I don't know how we could make stronger
126  * guarantees.
127  */

189     public static UserGroupInformation getEffectiveUser(KerberosConfig krb5Config)
190             throws IOException, LoginException, ConfigurationException {
191 
192         LOGGER.debug("config: {}", krb5Config);
193         UserGroupInformation user = UserGroupInformation.getCurrentUser();
194         if (UserGroupInformation.AuthenticationMethod.PROXY.equals(
195                 user.getAuthenticationMethod())) {
196             LOGGER.info("Current user is a proxy: {}", user);
197             user = user.getRealUser();
198         }
199 
200         // use current user or use Kerberos auth.
201         // note: loginUserFromSubject(lc.getSubject()) sets the loginUser, not new user
202         // note: getUGIFromSubject(lc.getSubject()) isn't sufficient.
203         LOCK.lock();
204         final String useSubjectCredsOnly = System.getProperty(USE_SUBJECT_CREDS_ONLY_PROPERTY);
205         try {
206             System.setProperty(USE_SUBJECT_CREDS_ONLY_PROPERTY, "false");
207             if (krb5Config.getClientPrincipal() != null) {
208 
209                 // Verify that Kerberos is enabled on the system. It should have been
210                 // initialized earlier - if we see this then the order in which methods
211                 // are called has been changed.
212                 // Note: since we don't have a Configuration object we can't initialize her.
213                 if (!UserGroupInformation.isSecurityEnabled()) {
214                     LOGGER.warn("A Kerberos account was provided but UGI security has not been " +
215                             "enabled");
216                 }
217 
218                 // this gives us better error messages for user but we don't use the results.
219                 LoginContext lc = getKerberosLoginContext(krb5Config, false);
220                 try {
221                     lc.login();
222                     LOGGER.debug("subject: {}", Arrays.toString(
223                             lc.getSubject().getPrincipals().toArray()));
224                 } finally {
225                     lc.logout();
226                 }
227 
228                 // get the actual user instance. We have to use this method instead of
229                 // using the LoginContext object since this method calls kinit (app). This
230                 // lets it get a Kerberos ticket with additional  properties not set by the
231                 // standard JAAS calls. (a tgt?) This allows the user to specify _HOST instead
232                 // of a specific system.
233                 user = UserGroupInformation.loginUserFromKeytabAndReturnUGI(
234                         krb5Config.getClientPrincipal(), krb5Config.getKeytabFile());
235             }
236         } finally {
237             if (useSubjectCredsOnly == null) {
238                 System.clearProperty(USE_SUBJECT_CREDS_ONLY_PROPERTY);
239             } else {
240                 System.setProperty(USE_SUBJECT_CREDS_ONLY_PROPERTY, useSubjectCredsOnly);
241             }
242             LOCK.unlock();
243         }
244 
245         // we always allow user impersonation.
246         if (krb5Config.getUserImpersonation()) {
247             user = UserGroupInformation.createProxyUser(krb5Config.getProxyUser(), user);
248         }
249 
250         LOGGER.debug("effective user: {}", user);
251 
252         return user;
253     }

265             throws LoginException, ConfigurationException {
266 
267         KerberosPrincipal principal = new KerberosPrincipal(krb5Config.getClientPrincipal());
268         File keytabFile = new File(krb5Config.getKeytabFile());

272         Set<Principal> principals = Collections.<Principal> singleton(principal);
273         Set<?> pubCredentials = Collections.emptySet();
274         Set<?> privCredentials = Collections.<Object> singleton(keytab);
275         Subject subject = new Subject(false, principals, pubCredentials, privCredentials);
276 
277         Map<String, String> params = new HashMap<>();
278         params.put(KERBEROS_PARAM_DO_NOT_PROMPT, Boolean.TRUE.toString());
279         params.put(KERBEROS_PARAM_USE_KEYTAB, Boolean.TRUE.toString());
280         params.put(KERBEROS_PARAM_KEYTAB, keytabFile.getAbsolutePath());
281         params.put(KERBEROS_PARAM_PRINCIPAL, principal.getName());
282 
283         // we don't need this since we explicitly add it ourselves. Keeping the
284         // commented code here so it's clear that we've made a deliberate decision
285         // to not set this to 'true'. Not setting it to 'false' either in case that

287         // params.put("storeKey", Boolean.TRUE.toString());
288

290         if (getTgt) {
291             params.put(KERBEROS_PARAM_USE_TICKET_CACHE, Boolean.TRUE.toString());
292             params.put(KERBEROS_PARAM_RENEW_TGT, Boolean.TRUE.toString());

296         params.put(KERBEROS_PARAM_REFRESH_KRB5_CONFIG, Boolean.TRUE.toString());

302                         for (Callback callback : callbacks) {

304                                 LOGGER.error(((TextOutputCallback) callback).getMessage());

306                         }
307                     }
308                 }, new CustomLoginConfiguration(
309                         Collections.singletonMap(serviceName, params)));
310 
311         return lc;

315      * Load KeyTab. getJaasDataSource() will store this value in the Subject's private
316      * credentials. getHadoopDataSource() only uses this method to verify that an appropriate
317      * keytab file was specified.
318      *
319      * @param keytabFile
320      * @param principal
321      * @return
322      * @throws ConfigurationException
323      */
324     public static KeyTab getKeyTab(File keytabFile, KerberosPrincipal principal)
325             throws ConfigurationException {
326 
327         if (!keytabFile.exists() || !keytabFile.canRead()) {
328             throw new ConfigurationException(ERROR_KEYTAB_FILE_DOES_NOT_EXIST_MESSAGE)
329                     .withReason(ERROR_KEYTAB_FILE_DOES_NOT_EXIST_REASON)
330                     .withReason(ERROR_KEYTAB_FILE_DOES_NOT_EXIST_RESOLUTION);
331         }
332 
333         // verify keytab file exists
334         KeyTab keytab = KeyTab.getInstance(principal, keytabFile);
335         if (!keytab.exists()) {
336             throw new ConfigurationException(ERROR_INVALID_KEYTAB_FILE_MESSAGE)
337                     .withReason(ERROR_INVALID_KEYTAB_FILE_REASON)
338                     .withReason(ERROR_INVALID_KEYTAB_FILE_RESOLUTION);
339         }
340 
341         // verify keytab file actually contains at least one key for this principal.
342         KerberosKey[] keys = keytab.getKeys(principal);
343         if (keys.length == 0) {
344             throw new ConfigurationException(ERROR_MISSING_KEYTAB_ENTRY_MESSAGE)
345                     .withReason(ERROR_MISSING_KEYTAB_ENTRY_REASON)
346                     .withReason(ERROR_MISSING_KEYTAB_ENTRY_RESOLUTION);
347         }
348 
349         // destroy keys since we don't need them, we just need to make sure they exist.
350         for (KerberosKey key : keys) {
351             try {
352                 key.destroy();
353             } catch (DestroyFailedException e) {
354                 LOGGER.debug(UNABLE_TO_DESTROY_KEY);
355             }
356         }
357 
358         return keytab;
359     }

534                 // Standard mode always uses account if it is specified in the Snap Configuration
535                 LOGGER.trace("standard mode - use credentials in account (doAs() user)");
536                 source = SourceOfKerberosCredentials.ACCOUNT;
537                 break;
538 
539             case EXTREME:
540             default:

542                 LOGGER.trace("fallback - use credentials on snaplex (LoginUser)");

544

549      * Ensure UserGroupInformation configuration is initialized.
550      *
551      * Note: we can eliminate the 'conf' parameter once we have HadoopService injected.
552      *
553      * @param conf Hadoop configuration.
554      */
555     public static void initializeUGI(Configuration conf) {
556         try {
557             // Force the UGI to initialize a loginUser object if the loginUser is not init yet.
558             // This is to make sure the loginUser is properly generated before we load the
559             // kerberos configurations into the UGI.
560             UserGroupInformation.getLoginUser();
561             LOGGER.info("Login User: {}, Current User: {}", UserGroupInformation.getLoginUser(),
562                     UserGroupInformation.getCurrentUser());
563         } catch (IOException e) {
564             LOGGER.warn("Fail to load the user information: {}", e);
565         }
566         if (!INIIIALIZED) {
567             CONF_LOCK.lock();
568             try {
569                 if (!INIIIALIZED) {
570                     LOGGER.info("(re)loading UGI configuration.");
571                     UserGroupInformation.setConfiguration(conf);
572                 }
573             } finally {
